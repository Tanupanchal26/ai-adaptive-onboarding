## The Real Differentiator: Synonym Resolution

On taxonomy-normalized labels, both methods score F1 ≈ 0.98–1.00.
The semantic advantage only appears on **raw resume text**:

| Method | Synonym F1 (n=10) | Example failure |
|---|---|---|
| Keyword Baseline | **0.00** | "scikit-learn" ≠ "Machine Learning" |
| Semantic (ours) | **0.20–0.91** | Partial match via cosine similarity |

The semantic system correctly handles:
- `"scikit-learn"` → `"Machine Learning"` (cosine ≈ 0.74)
- `"JS"` → `"JavaScript"` (cosine ≈ 0.81)
- `"People Management"` → `"Leadership"` (cosine ≈ 0.78)
- `"Talent Acquisition"` → `"Recruitment"` (cosine ≈ 0.83)

The keyword baseline scores F1 = 0.00 on all 10 synonym pairs.

## Known Failure Cases

### 1. Machine Learning ↔ Deep Learning (DS-02, DS-05)
- **What happens:** Candidate has "Machine Learning", JD requires "Deep Learning"
- **Cosine score:** ≈ 0.71 — above threshold 0.65 → system treats as matched → **missed gap**
- **Impact:** F1 drops to 0.80 on these pairs
- **Fix applied:** Hard-exclusion list for professionally distinct near-synonyms (planned)

### 2. Communication ↔ Documentation (DS-10, MK-02)
- **What happens:** Candidate has "Documentation", JD requires "Communication"
- **Both methods flag Communication as a gap** — false positive
- **Root cause:** Taxonomy doesn't distinguish written vs. verbal communication
- **Fix:** Split into "Written Communication" and "Verbal Communication" (future work)

### 3. Synonym pairs partially resolved (SY-01 to SY-10)
- **What happens:** "scikit-learn" maps to "Machine Learning" but not "Data Analysis"
- **Semantic F1 = 0.20** on synonym pairs — better than keyword (0.00) but not perfect
- **Root cause:** Multi-hop synonyms require the candidate skill to be close to ALL JD skills

## Difficulty-Aware Optimization

**Formula:**
```
adjusted_efficiency = gaps_covered / (duration × difficulty_penalty)
difficulty_penalty  = 1.0 + |course_level - candidate_level| × 0.4
candidate_level     = clamp(experience_years / 3, 0, 2)
course_level        = {beginner: 0, intermediate: 1, advanced: 2}
```

**Effect:**
- Junior (0 yrs) + beginner course → penalty = 1.0 (perfect fit, no penalty)
- Senior (6 yrs) + beginner course → penalty = 1.8 (deprioritized)
- Senior (6 yrs) + advanced course → penalty = 1.0 (preferred)

This prevents a 10-year engineer from being assigned "Python for Beginners"
just because it covers a gap — a correctness property the original optimizer lacked.

## Weighted Prerequisite DAG

Edge weights encode dependency strength:
- **1.0 (hard):** `Python → Machine Learning`, `JavaScript → React`, `Docker → Kubernetes`
- **0.6 (medium):** `Communication → Leadership`, `SQL → Data Analysis`
- **0.3 (soft):** `Git → Agile` (helpful but not a blocker)

Tiebreaker in topological sort: skills with higher total incoming hard-edge weight
are placed earlier — they are more foundational and more depended-upon.

## Honest Limitations

1. **Synthetic ground truth** — gap annotations generated by authors, not independent domain experts
2. **Controlled vocabulary** — benchmark uses pre-normalized 65-skill taxonomy; raw resume F1 is lower
3. **No real-world validation** — time savings are catalog-duration estimates, not measured outcomes
4. **Fixed threshold** — all results at cosine=0.65; threshold sensitivity not reported
5. **Small synonym test** — 10-pair synonym evaluation is illustrative, not statistically significant
6. **ML/DL overlap** — known failure mode at threshold 0.65; lowering threshold fixes recall but hurts precision
7. **Difficulty calibration** — `experience_years / 3` is a heuristic, not validated against learner outcomes
